# Планшетная адаптация: правила, подход и текущий статус

## Зачем это делаем
- У проекта исторически было 2 режима: mobile и desktop.
- Нужен полноценный 3-й режим: tablet.
- Цель: чтобы следующий разработчик/чат мог продолжать работу в едином стиле, без "прыжков" между подходами.

## Брейкпоинты (обязательно)
- `320-667` — mobile
- `668-990` — tablet
- `991+` — desktop

Источник:
- `utils/breakpoints.js`
- `styles/settings.scss` (`$bp-mobile-max`, `$bp-tablet-min`, `$bp-tablet-max`, `$bp-desktop-min`)

## Главный принцип tablet-версии
- На планшете открываем **PC-вариант компонентов/модалок/страниц**.
- Стили берем из PC, но размеры переводим в `tablet-vw(...)`.
- Шрифты используем через существующие миксины (`@include font...`), они уже внутри считают планшет через `tablet-font-vw(...)`.

## Техническая база

### SCSS / формулы
- Основной файл: `styles/settings.scss`
- Важное:
  - `@include tablet-only { ... }`
  - `@function tablet-vw($px)`
  - `@function tablet-font-vw($px)`
  - `@mixin font-three-mode(...)` и все `fontXX_YYY()`

### JS-ветвление по layout
- Не использовать legacy-порог `800` для новой логики.
- Использовать брейкпоинты из `utils/breakpoints.js`.
- Mobile layout должен включаться только на `<= 667`.

Пример корректного условия:
```js
const isMobileLayout = useMediaQuery(`screen and (max-width: ${BREAKPOINTS.mobileMax}px)`);
```

## Как делать новую tablet-адаптацию (пошагово)
1. Проверить, какой компонент рендерится на mobile/desktop.
2. Для tablet переключить рендер на PC-ветку (если сейчас уходит в mobile).
3. В соответствующем `...PC.scss` добавить/доработать блок `@include tablet-only`.
4. Перевести геометрию (width/height/margin/padding/radius/gap/line-height) через `tablet-vw`.
5. Шрифты оставить через `@include font...` (или явный `tablet-font-vw`, если это не миксин).
6. Проверить проблемные места:
   - вертикальное центрирование в кнопках (`line-height`, `display:flex`, `align-items:center`)
   - fixed heights, которые режут текст
   - вложенные селекторы `& div` vs `& > div` (часто ломают внутренние блоки)
7. Проверить на ширинах: `667`, `768`, `820`, `990`, `991`.

## Частые ошибки (и как не допускать)
- Ошибка: планшет попадает в mobile из-за legacy `matches` (800).
  - Решение: использовать `BREAKPOINTS.mobileMax` (`667`) для mobile-ветки.
- Ошибка: текст обрезается/исчезает после смены `span` -> `div`.
  - Решение: проверять родительские селекторы (`& div` может примениться к новому `div`).
- Ошибка: символы `+ / - / цена` в кнопках не по центру.
  - Решение: `display:flex; align-items:center; justify-content:center; padding:0; line-height:1;`
- Ошибка: clamp не работает.
  - Часто причиной является `display:flex` на текстовом элементе.

## Рекомендуемый формат работы для следующего агента
- Берем 1 блок/страницу за раз.
- Сначала фиксируем переключение layout (mobile/tablet/desktop).
- Затем переводим стили PC -> tablet.
- После каждого блока ручная проверка в devtools на ширинах `768` и `820`.
- Не смешивать глобальные рефакторы и точечные правки в одном шаге.
