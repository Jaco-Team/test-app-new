# Storybook FSD Structure

> **Архив / справка.** Активный дизайн-система и новый core — в `src/shared/ui` и `src/pages`.
> См. [TODO-NEW.md](../TODO-NEW.md). Новые UI-компоненты в `stories/` не добавляем.

Цель (историческая): перевести Storybook на `FSD`-организацию. Этот контур заменён слоем `src/shared/ui`.

## Базовые правила

- Новый контур историй: `stories/{app,pages,widgets,features,entities,shared}/**/*.stories.*`
- Весь старый набор историй и связанных компонентов вынесен в `stories/legacy/*`.
- Любые новые UI-истории создаем только в FSD-путях, а не в `legacy`.

## Назначение папок FSD

`stories/app`: истории app-уровня (глобальные провайдеры, layout, app-shell, инициализация темы/локали, интеграционные сценарии верхнего уровня).

`stories/pages`: истории полноценных страниц и route-сценариев (страница как композиция нескольких widgets/features).

`stories/widgets`: истории крупных самостоятельных блоков страницы (header, footer, модальные контейнеры, секции каталога, большие панели).

`stories/features`: истории пользовательских фич и действий (авторизация, фильтры, добавление в корзину, промокод, переключатели режимов).

`stories/entities`: истории сущностей домена (карточка товара, элемент заказа, карточка промокода, контакт, адрес и т.п.).

`stories/shared`: истории переиспользуемых примитивов и инфраструктуры UI (кнопки, инпуты, селекты, базовые элементы, общие обертки).

## Вне FSD-слоев (служебные папки)

`stories/legacy`: старые истории до миграции. Новые истории сюда не добавляем.

`stories/fixtures`: данные для сторис (mock/fixtures/json). Фикстуры должны быть переиспользуемыми и не зависеть от legacy-структуры.

## Правило зависимостей слоев

Допустимое направление: `app -> pages -> widgets -> features -> entities -> shared`.

Нижележащий слой не должен импортировать вышележащий.

## Текущее дерево

- `stories/app`
- `stories/pages`
- `stories/widgets`
- `stories/features`
- `stories/entities`
- `stories/shared`
- `stories/legacy`
- `stories/fixtures`
